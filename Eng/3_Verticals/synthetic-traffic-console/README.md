# Synthetic Traffic Console — the operator cockpit over the synthetic-traffic learning loop

> **Up:** [Verticals — the portfolio](../README.md) · **Russian counterpart:** [`Ru/3_Verticals/synthetic-traffic-console/README.md`](../../../Ru/3_Verticals/synthetic-traffic-console/README.md)

> **Part of the Real AGI portfolio.** A core-engine layer: the human-facing control room for the feedback/learning loop — where an operator drives synthetic users at an expert/chat under test, watches the runs, reads the evaluator's verdict, and feeds the lessons into the next cycle. **Stage:** spec + static mock (the module boundary and contracts are stabilized; runtime wiring is deliberately disabled).

A self-learning vertical needs a way to push controlled experience through itself and to read the consequences back. Synthetic Traffic Console is the operator-facing surface for exactly that: pick a target (an expert/chat under test), plan synthetic traffic (a mix of synthetic-user roles), set analysis goals, run a controlled cycle, watch live conversations and cost/time telemetry, read a completed-run report with evaluator scores, and act on learning recommendations before the next cycle. It is deliberately *only* the console — it owns the visible flow and the action requests, and it explicitly does **not** own the synthetic-user runtime, the traffic runner, the evaluator, or the learning attribution. It reaches those only through an explicit, versioned UI-to-runtime contract.

## How it realizes the concept

This is the **feedback organ** of the engine, made operable. In *agent = pipeline + feedback + self-learning*, this card is the cockpit over the last two terms: its seven-stage cycle (case setup → traffic setup → analysis goals → operational-cycle setup → live run → completed-run report → learning recommendations) *is* the labeled-consequence loop, with a human in the loop deciding what changes. Its engineering is a clean instance of the "LLM as an unreliable component" discipline the corpus argues for: the UI renders from a single typed view model and never infers runtime semantics from hidden internals; every action passes through a versioned contract, so a new action or lifecycle state is a contract change rather than a silent drift; and "no-runtime guardrails" keep real traffic and runtime wiring structurally off until an owner contract exists — judgment and execution held apart from the control surface by design. The domain it serves — the quality and evaluation of expert agents — is precisely what every vertical built on the constructor will need.

## What exists today

A contract-stabilized module, not a wired product. There is an interface spec defining what the console owns and must not own, a draft UI-to-runtime contract, a view-model schema with a validation check, a curated static fixture, and a static settings/console UI surface — all with runtime wiring and real traffic explicitly disabled. The module lives as a shared bounded context in two projects: its target home, the constructor platform ([expert-constructor-core](../expert-constructor-core/README.md)), and the legacy line it came from ([ai-test01](../ai-test01/README.md)). What does not exist yet: the live wiring to a real synthetic-user runtime, evaluator, and learning orchestrator — until that, the displayed run status, scores, and recommendations are rendering projections over a fixture, not live values.

## Where it could go

*Hypothesis:* this is the missing **observability-and-control layer** over the core's consequence loop. Self-learning ([essays-claude](../essays-claude/README.md)) supplies the decision → result → correction machinery; this console supplies the cockpit a human uses to aim it at a specific expert, run controlled cycles, and approve what changes before the next one. Wired to a real runtime and evaluator, it becomes the operator surface every vertical reuses to harden its expert against synthetic users before live ones — and a natural commercial product in its own right: a quality console for anyone shipping an expert agent.
